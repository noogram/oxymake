## 1. Why OxyMake Exists

Snakemake (2012) proved that file-based, rule-driven workflow orchestration
works. Twelve years and three thousand citations later its core ideas — backward
chaining, wildcard rules, incremental execution via file dependencies — remain
sound and generalize far beyond bioinformatics, to quantitative finance, ML
pipelines, data engineering, and any domain where reproducible computation
matters.

But Snakemake's implementation shows its age. Its Python DSL breaks static
analysis, IDE support, and deterministic parsing. Timestamp-only change
detection causes phantom re-runs after `git checkout`, NFS skew, or a stray
`cp`. DAG resolution takes minutes once jobs cross the thousand mark. Outputs
must be files: database tables, API calls, and virtual targets do not fit. The
architecture is monolithic, and the interface is for humans only — no
structured API for programmatic or agent-driven use.

The modern workflow landscape — Dagster, Prefect, Temporal, Flyte — has
innovated on scheduling, observability and state, but abandoned the simplicity
of file-based rules that made Make and Snakemake powerful in the first place.

OxyMake reclaims that simplicity with a modern engine.

## 2. The Founding Principle

> **Rust provides the engine. The workflow provides the intent.**

This principle, inspired by Gas Town's (Steve Yegge's agent-orchestration
rig) *Go provides transport, AI provides cognition*, draws a strict line of
responsibility through the system.

The **Rust engine** resolves the DAG, schedules jobs respecting topology and
resource constraints, computes content hashes for cache decisions, dispatches
to pluggable executor backends, emits structured events for every state
transition, persists execution state for crash recovery, and manages the
content-addressable cache.

The **workflow definition** declares rules with input/output contracts,
specifies wildcards and expansion sources, defines resource requirements,
chooses execution environments (conda, docker, uv, nix), places gates for
human-in-the-loop checkpoints, and selects the per-rule execution language
(shell, Python, R, …).

What the engine *never* does is also a load-bearing decision. It does not
decide a job is stalled based on elapsed time — it exposes the duration; the
observer (human or agent) decides. It does not guess whether a job should
re-run — the content hash says yes or no, deterministically. It does not infer
intent from patterns — the workflow declares intent explicitly and the engine
executes it faithfully. Same inputs + same rule + same parameters = same
caching decision, always, on any machine, at any time. **Determinism is not a
feature — it is the foundation.**

### 2.1 Language as Material

Each language in the stack is chosen as a *material* with specific properties
suited to its role, not as a general-purpose default. Rust is the right
material for the orchestration core: ownership, types, and zero-cost
abstractions make data races, null dereferences, and use-after-free
structurally impossible, and distinct types (`Rule` vs `ConcreteJob`) enforce
architectural invariants at compile time. TOML is the right material for the
workflow definition because it is declarative, statically parseable, and not
Turing-complete: any tool can read it without executing code, which is exactly
what enables IDE support, linting, and agent introspection. Python, R and
Julia are chosen by domain scientists for their ecosystems, and OxyMake
respects that by treating them as polyglot execution targets routed through
the `call` mode. Shell remains the universal glue for invoking existing tools
(`bwa`, `samtools`, `ffmpeg`).

## 3. Core Design Principles

### 3.1 Content-Addressable by Default

The source of truth for change detection is **file content**, not timestamps:

```
cache_key = blake3(
    rule_source_hash + sorted(input_content_hashes) + params_hash +
    env_spec_hash + platform + trust_scope
)
```

All dimensions must be present from day 1 (OX-1, see §6). Timestamps serve as
a fast-path optimization: when mtime and size are unchanged since the last
recorded run we skip the hash; when mtime differs we hash to confirm. This
eliminates phantom re-runs from `git checkout`, `cp`, NFS clock skew, CI fresh
clones, and stray `touch`. The cache is shareable across machines — same
content hash = same result, regardless of where or when it was computed.

### 3.2 Daemon-Free by Default

No daemon. No coordinator server. Each `ox run` is a self-contained process
that resolves the DAG, executes, writes state to `.oxymake/state.db`, and
exits. Multiple `ox run` processes coordinate via atomic SQLite operations
(OX-11) — no central orchestrator needed. For gates, the pattern is
checkpoint-and-resume: when the scheduler hits a gate it completes pre-gate
jobs, persists state, and exits with code 75; a `ox gate approve <id>`
followed by `ox run` resumes from the checkpoint. An optional `ox serve`
daemon mode is available for long-running orchestration, but never required.

### 3.3 API = CLI

The CLI is a thin wrapper over `oxymake-api`. Every operation is available
both as a Rust function call and as a CLI command with `--json` output. When
`--json` is active, events are emitted as newline-delimited JSON on stdout —
an agent reads line-by-line and reacts to typed events:

```jsonl
{"type":"run.started","total_jobs":103429,"to_run":847,"cached":102582}
{"type":"job.completed","id":"align_S001","duration_ms":272000}
{"type":"gate.reached","id":"qc_check","message":"Review QC metrics"}
```

For programmatic Rust embedding, the scheduler exposes a
`tokio::broadcast::Receiver<Event>` channel.

### 3.4 Scaling Ladder

Same workflow file. Different `--executor` flag. The scheduler is always local
(star topology — single coordinator, never distributed).

| Level | Flag | What happens |
|-------|------|-------------|
| Sequential | (default) | `std::process::Command`, one job at a time |
| Local parallel | `-j N` | tokio pool, N concurrent jobs |
| SLURM | `--executor slurm` | `sbatch` submission, `sacct` polling |
| Ray | `--executor ray` | Ray Jobs API submission |
| Kubernetes | `--executor k8s` | K8s Job resource creation via kube-rs |

The transition from laptop to cluster requires zero workflow changes.

### 3.5 Declarative Workflow, Polyglot Execution

The workflow is always TOML. Individual rules execute in any language through
four modes forming a **spectrum from opaque to optimizable**: `shell` (opaque
command, files only), `run` (inline script, files only), `script` (external
script file, files only), and `call` (pure function, files OR memory). Moving
up the spectrum, OxyMake gains optimization power (in-memory passing, type
checking, automatic serialization) but the author gives up I/O control. The
natural migration path is to start with `run` for rapid prototyping and
migrate to `call` once the function stabilizes.

The `call` mode is the key innovation. The function is pure — it receives
objects, returns objects — and OxyMake manages all I/O outside the function.
In **file mode** OxyMake reads the input with the declared format codec, calls
the function, and writes the result with the declared codec. In **memory
mode**, the upstream job hands its DataFrame directly to the downstream job
through the same in-process worker — zero disk I/O, zero serialization. The
function never knows the difference.

### 3.6 Materialization Policy

`call`-mode outputs carry a `materialize` field: `always` (the default —
write to disk, cacheable), `auto` (materialize only if a non-`call` downstream
needs the file), `never` (memory only, lost on process death, not cached), or
`final` (materialize only if this output is a DAG leaf). A global CLI override
switches the entire workflow between modes: `ox run --materialize=final` for
exploration, `ox run --materialize=never` for a speed test, `ox run` for
production. The trade-off is explicit: speed vs reproducibility.
Non-materialized outputs are not cached.

### 3.7 Plugin Architecture

Six plugin axes, each defined by a Rust trait, decouple `ox-core` from every
substrate it touches: **Executor** (local, SLURM, K8s, Ray), **Storage**
(local FS, S3, GCS, SSH), **Environment** (system, uv, conda, docker, nix,
apptainer), **Reporter** (terminal, NDJSON, GitHub Actions, webhook),
**FormatCodec** (parquet, csv, json, arrow, pickle), and **StateBackend**
(SQLite, Dolt, PostgreSQL). The default `SqliteBackend` preserves the
daemon-free single-file model. Alternative backends (e.g. Dolt for
version-controlled state with time-travel queries) are selected at runtime
without changing the workflow.

Plugin loading is compile-time via Cargo feature flags. The minimal binary
(local executor + local storage + system environment + terminal reporter +
basic codecs + SQLite state) has minimal dependencies and compiles fast. Heavy
plugins (K8s, S3, Dolt) are opt-in features. The crate dependency graph
itself is a tested invariant (OX-3 — separable axes).

### 3.8 Errors as First-Class Design

Error messages trace the full causal chain through the DAG. Every error
answers three questions: what failed, why, and how to fix it. In JSON mode
the same structure is machine-parseable — agents can programmatically read
the chain, identify the root cause, and take corrective action without
parsing human text.

## 4. Tags, Hierarchical Views, and Organic Growth

Every workflow dimension is filterable through tags. Tags operate on two
levels: **implicit** (every resolved wildcard automatically becomes a tag —
a job `align(cohort=human, sample=NA12878, reference=hg38)` has those tags
without any declaration) and **explicit** (declared per-rule for cross-cutting
concerns: `tags = { stage = "features", domain = "genomics", cost = "low" }`).

**Filtering with `--where`** selects target jobs, then backward-chains to
include all necessary upstream dependencies:

```bash
ox run --where cohort=human --where stage=qc
ox run --rule '/^qc.*/' --where 'sample=~/sample_.*/'
```

The `--where` clause is a *leaf filter* — it selects which terminal jobs to
target; the backward-chaining algorithm pulls in exactly the upstream jobs
needed. This is more powerful than Snakemake's target-file approach because
it works across wildcard dimensions without knowing specific file paths.
`--rule` filters by rule name (exact or regex). Both compose with AND
semantics.

**Conditional guards** (`when = "sample in @candidate_samples"`) enable
non-uniform DAG branches without phantom nodes: a job whose guard evaluates
to false is never created. Guards are evaluated at DAG resolution time, not
at execution time, so the resulting DAG is genuinely non-uniform.

**Semantic zoom** uses tags to collapse jobs into meta-nodes for display.
`ox dag --group-by stage` shows the pipeline view; `--group-by stage,cohort`
shows a two-dimensional grid; `--where cohort=human` zooms into individual
jobs. Grouping is a view transformation, not a separate data structure —
the underlying DAG is unchanged.

**Organic growth** is the operating mode: the content-addressable cache means
adding rules never invalidates existing results (week 1's 50 features are
still cached at week 6), the include system splits the workflow across files,
outputs from run N are automatically available as inputs to rules added in
run N+1, snapshots (`ox snapshot create "baseline-v1"`) capture meaningful
research milestones with instant diffing, and `--note` annotations on every
run turn `.oxymake/state.db` into a lightweight, persistent lab notebook.
OxyMake is designed for workflows that *grow with understanding*, not for
"define a complete pipeline, then run it".

## 5. Architecture

### 5.1 Crate Structure

The workspace is split into single-responsibility crates: `ox-core` (DAG,
scheduler, traits, types, events), `ox-format` (TOML parsing), `ox-state`
(SQLite persistence), `ox-cache` (content-addressable cache), `ox-plan`
(optimization passes), `ox-api` (public Rust facade), then one crate per
plugin variant (`ox-exec-{local,slurm,ray,k8s}`,
`ox-storage-{local,s3}`, `ox-env-{system,uv,conda,docker,nix}`,
`ox-codec-{core,arrow,pickle}`, `ox-report-{term,json}`) and finally
`ox-cli` as a thin clap wrapper over `ox-api`. Responsibility boundaries are
strict and tested (OX-3): `ox-core` never does file I/O, `ox-format` never
executes rules, `ox-state` never decides if a job should re-run, `ox-cli`
never contains business logic.

### 5.2 The Three Graphs

OxyMake uses three distinct graph representations, each at a different
abstraction level, following the pattern proven by DataFusion (logical →
physical), Bazel (target → action), and Spark (RDD lineage → stages →
tasks).

**RuleGraph** — the *logical* graph. Nodes are rules with unresolved
wildcards; edges are input/output pattern dependencies between rules. The
workflow as the user writes it: abstract and compact, one `align` node
represents *all* alignment instances. Built by `ox-format` from the
Oxymakefile. Cycle detection, ambiguity detection (OX-5), and pipeline-level
visualization operate here.

**JobGraph** — the *physical* graph. Nodes are `ConcreteJob`s with
wildcards resolved plus `OutputRef`s (files, memory, virtual); edges are
bipartite Produces/Consumes relations. The RuleGraph expanded (wildcards
resolved, guards evaluated) then optimized through a series of composable
passes: cache pruning, task fusion, group scheduling, materialization
elimination, critical path analysis, partition planning. Each pass is a
`trait OptimizationPass` that takes ownership of the graph and returns a
transformed graph plus a `PassResult`.

**ExecGraph** — the *runtime* graph. The JobGraph annotated with live
status, metrics (wall_time, peak_mem), and log paths. Scheduling, monitoring,
crash recovery, and dynamic re-resolution (when a checkpoint job completes
and the JobGraph grows — OX-2) all happen at this level.

`ox plan` is the inspector — like SQL's `EXPLAIN` or Bazel's `aquery`, it
exposes each level: `--level=rules`, `--level=jobs --no-optimize`, default
(optimized), `--show-pass=task_fusion`, and `--json` for agents.

### 5.3 Core Data Model

The `Rule` carries a name, optional priority, input/output patterns (with
format and named keys for `call` mode), an `ExecutionBlock` (`Shell | Run |
Script | Call`), resources, optional environment spec, explicit tags, optional
wildcard constraints, an optional `when` guard, expand mode (`Product` or
`Zip`), error strategy (`Terminate | Retry { count, backoff } | Ignore |
Finish`), optional timeout, optional executor override, and log config. The
`ConcreteJob` is the resolved instance: same fields with wildcards bound to
values. `OutputRef = File(PathBuf) | Virtual { id, check } | InMemory {
type_hint }`. `MaterializePolicy = Always | Auto | Never | Final`. The
`Event` enum covers the full lifecycle: `RunStarted, JobQueued, JobStarted,
JobCompleted, JobFailed, JobSkipped, GateReached, GateApproved, RunCompleted,
RunFailed`.

### 5.4 Executor Trait

The `Executor` trait covers init, health check, capability query
(`supports_gpu`, `supports_streaming`, `supports_shadow_dirs`,
`supports_memory_passing`, `max_timeout`), workspace prepare/finalize,
execute, cancel, poll_status, and max_concurrency. `supports_memory_passing`
is true for `LocalExecutor` (same process), false for Slurm/K8s (separate
processes — the scheduler auto-promotes `InMemory` outputs to `File`). The
`prepare_workspace`/`finalize_workspace` lifecycle enables shadow dirs,
hermetic isolation, and remote staging.

### 5.5 State Management

Three logically separated state concerns share a physical store: **execution
state** (ephemeral, reconstructible from DAG + cache — what is running right
now, session tracking for cooperative multi-process execution), **cache
state** (persistent, immutable, content-addressed under
`.oxymake/cache/{hash_prefix}/{hash}`, independent of SQLite — can be shared,
synced, or rebuilt), and **audit state** (append-only, never pruned, the
provenance record). All three are accessed through the 22-method
`StateBackend` trait. The default `SqliteBackend` uses `rusqlite` with WAL
mode and atomic job claiming; alternative backends can provide
version-controlled state (Dolt), team-shared state (PostgreSQL), or
cloud-native state (DynamoDB) without changing the scheduler.

## 6. Named Invariants (OX-\*)

Every architectural commitment that is **extremely costly to change later**
carries a stable identifier in a single `OX-\*` namespace shared by the
thesis, ADR-015 *Named Invariants and TLA+ Scope*, the TLA+ specs in
`spec/tla/`, and the corresponding Rust tests in `crates/*/tests/`. A
contributor encounters the same identifier in code, spec, ADR, and prose.

The CSTAFP surfaces — *Concurrent State Transitions Across independently-
Failing Peers* — are defended by TLA+ modules; the structural and
declarative surfaces are defended by `cargo test` and CI gates. ADR-015 is
the load-bearing reference for the TLA+ scope and sunset mechanism.

| **OX-18** | ReproducibilityLockfile | `ox.lock` (committed to git) captures `ox_version`, platform, workflow hash, environment hashes, and optimized JobGraph hash; `ox run --locked` refuses to run if the plan differs | `crates/ox-api/tests/lockfile_*.rs` |

The OX-\* namespace is **the** vocabulary for cross-referencing architecture
between the thesis, ADRs, TLA+ modules, Rust tests, CI gates, and design
notes. A future invariant is added with the next free index, named in
PascalCase, anchored in at least one of (Rust test, TLA+ spec, CI gate),
and back-referenced from the ADR that ratified it. There is no parallel
numbering (no `INV-N` or `§6.N`) — the cost of two name spaces for the
same concept is the most expensive form of documentation rot we have
identified.

## 7. Open Design Questions

A short list of issues identified during the final audit that need
resolution during implementation. Each is specified enough to implement
but requires detailed decisions on the way down.

**SQLite WAL on network filesystems.** WAL does not work on NFS/Lustre/GPFS.
In Phase 1 (local only) this is a non-issue. In Phase 2 (distributed),
`.oxymake/state.db` MUST be on a local filesystem: the scheduler runs on the
submission node, compute nodes only receive job specs. `ox run --executor
slurm` validates the location and errors clearly if violated.

**Scatter/gather — dynamic output count.** A rule with `output = [{ pattern
= "chunks/{sample}/chr_{chunk}.bam", dynamic = true }]` is a checkpoint: the
scheduler executes it, then re-resolves the DAG by globbing actual outputs.
A downstream rule with `input = [{ gather = "results/{sample}/chr_{chunk}.vcf",
from = "split_chromosomes" }]` collects them. Designed into the resolver from
day 1 (OX-2).

**Include system.** Included files merge their `[config]` with the root
(root wins on conflict, or explicit `override = true`). Rule names are
globally unique across all includes. Circular includes are detected and
error out. Future: namespaced includes (`{ path = "lib/stats.toml",
namespace = "stats" }`) producing `stats.normalize`, `stats.zscore`.

**Resource-aware local scheduling.** `-j N` caps concurrent jobs; resource
declarations (`cpu`, `mem`, `gpu`) further constrain dispatch. The scheduler
maintains a budget and only dispatches jobs that fit — otherwise `ox run
-j 8` on a 32 GB machine with eight 16 GB jobs OOMs.

**Secrets in command lines.** OX-7 keeps secrets out of logs and SQLite,
but `shell = "curl -H 'Authorization: {params.api_key}'"` exposes them in
`ps aux` and SLURM's `sacct`. Resolution: secrets are passed as env vars,
never interpolated into command strings: `secrets = { API_KEY =
"$VAULT_API_KEY" }`.

**TOCTOU mitigations.** Three risks: stale-session reclaim (two-phase
reclaim with one heartbeat delay), cache-invalidation race (validate inside
a transaction before marking skipped), and output-write race (impossible by
construction — OX-11 + OX-5).

**Operational commands.** `ox logs <job-id>`, `ox clean --cache`, `ox gc
--max-cache-size 10G`, `ox rerun <job-id> --cascade` are missing from the
v0.1 surface and need design.

**Format versioning.** `ox_version = "0.1"` in every Oxymakefile;
compatibility validated at parse time with migration guidance when the
format evolves.

## 8. Target Use Cases

**Bioinformatics.** The original Snakemake domain — genomics, proteomics,
transcriptomics workflows with hundreds of samples passing through
alignment, QC, variant calling, annotation. Transition: `ox run -j 8` on
laptop, `ox run --executor slurm` on cluster, same workflow.

**Quantitative finance research.** Alpha research pipelines: data ingestion,
feature engineering, model training, backtesting, performance attribution.
Reproducibility is critical for regulatory and research integrity; the
audit trail (OX-9) and lockfile (OX-18) make every result reconstructible.

**ML training.** Data preprocessing, feature engineering, training,
evaluation, deployment. Per-rule resource declarations and per-rule
container environments make GPU jobs and CPU jobs coexist in the same DAG;
gates check metrics before deploy.

**Data engineering.** ETL pipelines, warehouse loading, report generation.
Virtual outputs (`{ type = "virtual", id = "api://salesforce/accounts" }`)
let the DAG depend on database tables and external APIs as first-class
nodes.

**Agent-driven workflows.** An agent invokes `ox run --json` and consumes
the event stream: on `gate.reached` it reviews and approves; on `job.failed`
it reads the structured error chain, adjusts parameters, and retries; on
`run.completed` it harvests the outputs. The same loop works from the Rust
API directly through `tokio::broadcast::Receiver<Event>`.

---

## 9. The UX Contract

OxyMake must be a tool people **love** using. This is not optional.

**Speed is the foundation.** Workflow parse < 10 ms (must feel like opening
a file). DAG resolution for 100 k jobs < 1 s (`--dry-run` must be instant;
10 k is measured at 64 ms, the 100 k target is projected, not yet benchmarked).
Incremental check when nothing changed < 100 ms (must feel like "nothing to
do" is free). First job dispatch < 200 ms (no perceptible delay between
command and action). These are requirements, not aspirations.

**Progressive disclosure.** `ox run` for the beginner; `ox run -j 8` for
the intermediate; `ox run results/sample_A.bam results/sample_B.bam` for
targeting; `ox run --executor slurm --set resources.align.mem=32G --profile
hpc` for the expert; `ox run --json | jq …` for the power user.

**CLI aesthetics.** Clean alignment. Time context on every job. Progress
visible at a glance. `stderr` for progress, `stdout` for data when piped.
Honor `NO_COLOR` and `--color never`. `-h` for brief help, `--help` for
full documentation. No emoji.

**Error messages** answer three questions every time: what failed, why, and
how to fix it. No error leaves the user or agent without a next step.

## 10. Distributed Compute and Environments

Five executors share the `Executor` trait — submit, monitor, collect,
cancel, cleanup. **SLURM** is the primary HPC backend (`sbatch` + `sacct`
polling, resource mapping to `--cpus-per-task`, `--mem`, `--time`,
`--gres=gpu:N`). **Ray** for Python-heavy workloads (Ray Jobs API).
**Kubernetes** for cloud-native deployments (Job resources via kube-rs).
**Local** for the laptop. **Ray serve / future** for streaming. Cluster and
cloud configuration live in profiles (`profiles/slurm-hpc.toml`), never in
the workflow.

Six environment providers cover the spectrum: `system` (no isolation), `uv`
(Python — first-class, the modern packaging standard, OxyMake calls
`uv run` with the right project), `conda` (multi-language scientific),
`docker` and `apptainer` (container isolation, the latter HPC-friendly and
rootless), and `nix` (hermetic, reproducible). Environments are prepared
once and cached: the content hash of the spec is the cache key.

---

## 11. Relationship to Agent Orchestrators

An agent orchestrator and OxyMake are **complementary**, not competing: the
orchestrator drives agents (reasoning, deciding); OxyMake drives computations
(deterministic, reproducible). The orchestrator's state store is built for
coordination (Gas Town: beads in Dolt — transactional, distributed); OxyMake's
is built for speed (SQLite + CAS cache — local, fast).

Two orchestrators have actually driven this project: development began piloted
by Gas Town (Steve Yegge's rig — polecats, refinery, beads), and is driven
today by cosmon (molecules, formulas, fleets). The full account is in
[docs/MAKING-OF.md](docs/MAKING-OF.md).

**What exists on the OxyMake side** — shipped and tested: an autonomous agent
can invoke `ox run`; the orchestrator can inspect `ox status --json`; every run
emits an NDJSON event stream (`ox run --json`); gates can be approved
programmatically (`ox gate approve <gate> --approver "agent:…"`).

**Design targets on the orchestrator side** — intended, not yet built: gate
approvals issued by orchestrator agents as part of a fleet policy; an
orchestrator work item (a molecule of type `dag`) wrapping an OxyMake workflow;
OxyMake's event stream feeding the orchestrator's event journal.

## 12. Implementation Phases

**Phase 1 — Local MVP.** `ox-core` (DAG, scheduler, wildcards, events,
traits), `ox-format`, `ox-cache` (blake3, OX-1), `ox-state` (SQLite,
OX-6/OX-8/OX-9/OX-11), `ox-exec-local`, `ox-storage-local`, `ox-env-system`,
`ox-env-uv`, `ox-report-term`, `ox-report-json`, `ox-api`, `ox-plan` (cache
pruning, task fusion, materialization elimination), and `ox-cli` (`run`,
`cancel`, `invalidate`, `plan`, `status`, `dag --group-by`, `snapshot`,
`history`, `explain`, `init`, `lint`). Deliverable: usable for local
workflows, 10–100× faster DAG resolution than Snakemake, beautiful CLI,
agent-friendly JSON.

**Phase 2 — Distributed + Gates.** `ox-exec-slurm`, `ox-exec-k8s`,
`ox-storage-s3`, gates/checkpoints, crash recovery (reconcile with executor
state on restart), remote cache sharing, resource-aware scheduling,
`ox-env-conda`, `ox-env-docker`. Deliverable: same workflow runs on laptop
and HPC cluster.

**Phase 3 — Ecosystem.** `ox-exec-ray`, `ox-env-nix`, `ox-env-apptainer`,
WASM plugin system, LSP for Oxymakefile.toml, rule registry, `ox serve`
daemon mode, Snakefile compatibility layer, web dashboard (optional, separate
project), CI/CD integrations.

---

## 13. Non-Goals

OxyMake explicitly does NOT aim to replace general-purpose task runners
(use `just`), to be a CI/CD system (use GitHub Actions, GitLab CI), to be
a data catalog or lineage system (use Datahub, OpenLineage), to be an agent
orchestrator (that is the complement's job — Gas Town, cosmon; see §11), to
embed a general-purpose programming
language in workflow definitions, or to support interactive/notebook-style
execution (use Jupyter).

## 14. Open-Source Readiness

If the project achieves quality and coherence, it will be released as
open-source. From day 1: dual Apache-2.0/MIT license (Rust ecosystem
standard); a `CHANGELOG.md` summarizing user-facing changes per PR;
semantic versioning (0.x during development, 1.0 when the API stabilizes);
GitHub Actions CI with test, coverage, lint, format, clippy, and the
OX-\* gates (separable axes, forbid-pyo3, exactly-once terminal, …); a
contributing guide for rules, plugins, and optimization passes; the
Contributor Covenant code of conduct; issue templates for bugs, features,
and RFC proposals; and a clean `git clone` that builds with no internal
dependencies.
