# ADR-016: Metrics as Single Source of Truth

## Status
Accepted (drafted from the metrics deliberation §3 T2 + §5 / Task 1 sibling ADR ;
companion to a one-shot reconciliation task).

## Metadata

- **Date:** 2026-05-28
- **Kind:** `decision`
- **Family:** `EXT`
- **Supersedes:** `none`

## Context

OxyMake's public surfaces — `README.md`, `docs/paper/oxymake-paper.tex`,
`bench/snakemake-vs-oxymake/RESULTS.md`, `docs/paper/experiment-results.md`
— each carry quantitative claims about the same underlying reality (DAG
resolution latency, SLOC, test counts, crate counts, throughput). Until
this ADR, each surface owned its own copy of those numbers. The numbers
drifted.

The drift was not hypothetical. It was load-bearing in the metrics
deliberation §1.1 (kahneman, Type-2) and §1.4 (torvalds, code
taste) — the same metric **DAG resolution at 10K jobs** appeared as
**62 ms** in `README.md:16`, **102 ms** in
`docs/paper/experiment-results.md:30`, and **688 ms** (median) in
`benchmark/perf/results.md:16` of an earlier benchmark snapshot. Three
surfaces, three numbers, one reality. A quant reading the project
crosses two of three surfaces before reaching for `cargo install` ; the
trust budget collapses at minute 8.

This is the **second recurrence** of the same structural pathology. The
parent délib (parent §3 T2) already inscribed the pattern at the
**LaTeX-macro level** — paper `\metric…` macros vs `Table~\ref{tab:…}`
prose vs raw `cargo test` output disagreed on test counts. That
recurrence was closed by introducing `docs/paper/metrics.tex`,
regenerated by `docs/paper/scripts/regenerate-metrics.sh` and pulled in
via `\input{metrics}` from `oxymake-paper.tex`. The mechanism worked
**within the paper**. It did not extend to the README or the benchmark.

When the same structural pathology surfaces twice at two different
scales, a local patch is insufficient. The ADR generalises the
single-source pattern that already works for the paper to all surfaces
that quote project metrics.

### Sources of truth that currently exist

| Surface                                       | Mechanism                                                                  | Status                                  |
| --------------------------------------------- | -------------------------------------------------------------------------- | --------------------------------------- |
| `docs/paper/metrics.tex`                      | regenerated by `docs/paper/scripts/regenerate-metrics.sh` on every `make`  | works **for paper only**                |
| `README.md`                                   | hand-edited prose                                                          | drifts silently                         |
| `docs/paper/experiment-results.md`            | hand-edited prose                                                          | drifts silently                         |
| `bench/snakemake-vs-oxymake/RESULTS.md`       | regenerated by `bench/snakemake-vs-oxymake/render_results.py` from `measurements.tsv` | works **for headline tables only**, headline-claim values do not propagate back to the paper or README |

The four sources update on four different cadences and disagree.

### What stays out of scope

TLA+ specifications (`spec/tla/*.tla`) intentionally **do not** carry
performance numbers ; they encode behavioural invariants (cache
consistency, cooperative claim, cancel propagation). Performance is a
measured quantity, not a specified invariant. This ADR does not push
metrics into the TLA+ corpus. If a future ADR wires a performance bound
into a TLA+ refinement check, that ADR will name the bound and the
canonical source it pulls from.

## Decision

**There is exactly one canonical metrics file. Every other surface that
quotes a project metric is a consumer that reads from that file at build
time.**

- **Producer.** A single workspace-level script,
  `scripts/regenerate-metrics.sh`, populates a JSON file
  `metrics/metrics.json` from live workspace measurements. Each metric
  is a top-level key with a numeric `value`, a `unit`, and a `source`
  field naming the measurement command. Benchmark perf numbers (e.g.
  `dag_resolution_10k_ms`) are written by the bench harness ; structural
  counts (SLOC, tests, crates, commits) are written by the workspace
  script.

- **Consumers.** Each downstream surface owns a thin renderer that reads
  `metrics/metrics.json` and emits the consuming artefact's expected
  form :

  | Consumer                                                  | Renderer                                                   | Output                                                  |
  | --------------------------------------------------------- | ---------------------------------------------------------- | ------------------------------------------------------- |
  | `docs/paper/oxymake-paper.tex` (via `\input{metrics}`)    | `docs/paper/scripts/regenerate-metrics.sh`                 | `docs/paper/metrics.tex` (LaTeX `\newcommand` macros)   |
  | `README.md`                                               | `scripts/regenerate-readme-metrics.sh`                     | rewrites tagged anchor blocks in `README.md` in place   |
  | `bench/snakemake-vs-oxymake/RESULTS.md`                   | `bench/snakemake-vs-oxymake/render_results.py` (existing)  | Markdown table from `measurements.tsv` (headline propagates to `metrics/metrics.json` as a side-effect) |

- **Tagged anchors in Markdown.** README and other Markdown surfaces
  delimit metric-driven regions with HTML comments :

  ```markdown
  <!-- METRICS:BEGIN dag_resolution_10k -->
  10K-job DAG resolved in 62ms
  <!-- METRICS:END -->
  ```

  Renderers rewrite the content between the markers and **only** that
  content. Prose around the markers is operator-owned and never
  touched.

- **Build pipeline.** A top-level `Makefile` carries a `metrics` target
  that orchestrates the producer + every renderer. Each consumer's
  existing build path (`make` in `docs/paper`, `run.sh` in the
  benchmark) depends on `metrics/metrics.json` so the file is
  regenerated before any artefact that quotes it is rebuilt. CI carries
  a `metrics-check` target that re-runs the producer in a clean tree
  and fails if the result differs from the committed
  `metrics/metrics.json` — same gate shape as `cargo fmt --check`.

- **Provenance.** Each value in `metrics/metrics.json` carries
  `source` (the command), `measured_at` (UTC date), and where relevant
  `hardware` (CPU, RAM, OS) so the canonical file is self-describing.
  A quant reading the JSON can re-derive any number on their machine
  without grepping for the source command.

- **Reconciliation.** The companion reconciliation task reconciles
  the existing 62/102/688 ms triple-drift to a single canonical number
  by re-running the measurement and committing the value into
  `metrics/metrics.json`. This ADR establishes the structural shape so
  the reconciliation sticks ; the one-shot is the **first instance** of
  the discipline, not the proof of it.

## Consequences

**Easier.**

- A reader can answer *« what does OxyMake actually measure? »* in one
  file. No more crossing three surfaces and finding three numbers.
- Adding a new public surface (a marketing page, a slide deck, a paper
  appendix) costs one new renderer, not a re-grep of every existing
  surface.
- CI catches drift mechanically. The `metrics-check` gate prevents the
  pathology from reopening through inattention — a contributor who
  hand-edits a metric value in the README without re-running the
  renderer will see a CI failure with a diff that names exactly which
  metric drifted from which canonical source.
- The discipline scales without bureaucracy. The producer is one
  script ; each consumer is a thin renderer ; the contract between
  them is one JSON file with a documented schema.

**Harder.**

- Editing a metric value in README requires re-running a renderer
  rather than typing a number. The trade is worth it ; a one-line `make
  metrics-readme` is cheaper than the alternative (silent drift,
  délibérations, ADRs).
- New metrics require deciding *« which surface produces this, which
  consume it »*. This is good — the question is structural and was
  invisible before. It surfaces a design discipline rather than
  creating one.
- The benchmark harness becomes a producer (writes
  `dag_resolution_10k_ms` into `metrics/metrics.json` after a
  measurement run) in addition to its existing role of rendering
  `RESULTS.md`. The split is mechanical and the harness already does
  the work ; what changes is the persistence target.

**What this ADR does not promise.**

- It does not promise that every metric in the workspace flows through
  `metrics/metrics.json`. Internal observability metrics (rot signals,
  patrol metrics, runtime traces) stay under their own paths
  (`docs/health/`, `.cosmon/state/patrol-metrics.json`,
  `.cosmon/state/runtime-trace.jsonl`). The single source applies to
  **metrics quoted in public-facing artefacts** (README, paper,
  benchmark RESULTS, doc-tree narratives).

- It does not promise that the producer script ships fully populated
  with every historical metric on the day this ADR lands. The minimum
  viable scaffolding is what lands here ; subsequent waves migrate
  individual metrics into the canonical file as they are touched. Each
  migration is reversible ; each removal of a duplicate is auditable in
  `git log`.

- It does not promise zero-cost CI. The `metrics-check` step
  re-executes measurement commands ; on a cold workspace this can take
  minutes. The cost is amortised against the cost of the pathology it
  prevents — and it runs only when files that produce metrics change.

## Alternatives Considered

### Alt 1 — One-shot reconciliation only, no structural fix

*Pick the canonical number once, hand-edit the three surfaces, move on.*

This is **the companion task** considered alone.
Rejected as the **sole** fix because the parent délib already inscribed
this pattern at one level (LaTeX macros) and it recurred at a different
level (README/paper/bench) within weeks. A second occurrence by the
same operator, on the same project, with the same proximate cause
(*hand-edited metric drifted*) is the structural signal the délib
called out. Companion is necessary ; ADR is necessary ; neither alone
is sufficient.

### Alt 2 — Push metrics into a Rust crate (`ox-metrics`)

*Generate a Rust constant, embed it in code, derive consumed artefacts
from the binary.*

`ox-metrics` already exists as an **execution-metrics collection** crate
(runtime telemetry). Co-locating documentation metrics (SLOC, test
count) into the same crate would conflate two unrelated concerns :
**what the system measures about itself at run-time** vs **what the
project measures about its own source code at build-time**. The two
have different consumers, different update cadences, and different
trust models. Rejected.

### Alt 3 — Push metrics into LaTeX macros, drop JSON

*Make `metrics.tex` the canonical file ; have README pull from it.*

`metrics.tex` is paper-formatted (`58{,}966` with LaTeX thousands group,
`\newcommand{\metricSLOC}{…}` syntax). README would need to strip the
LaTeX wrapping ; a third surface (slides, web) would need its own
parser. The JSON intermediate is unifying because it is **
representation-neutral** ; the LaTeX form is **representation-coupled**
to the paper. Rejected.

### Alt 4 — Push metrics into a database (Dolt, SQLite)

*Treat metrics as data, query at build time.*

Considered and rejected for the same reason ADR-004 picked SQLite over
Dolt : the surface area must match the problem. Project metrics are
~30 numbers updated by build scripts ; a JSON file in the repository
satisfies the read pattern, the write pattern, and the auditability
pattern without introducing a runtime dependency on a database. Rejected.

### Alt 5 — Do nothing, accept drift

*Acknowledge that public surfaces drift, document the policy of
*« the most recently regenerated surface wins »*.*

Rejected as the false-elegance option. Drift in a quant project erodes
trust at minute 8 of first contact — earlier than any code path Sarah
might reach. The cost of doing nothing is paid by every reader, every
time ; the cost of this ADR is paid once.

## Sunset

This ADR is not in conflict with any prior ADR. It generalises the
local pattern that `docs/paper/scripts/regenerate-metrics.sh` already
implements ; the prior pattern is preserved (the paper renderer still
exists and still emits `metrics.tex`), only its **input** changes (it
reads from `metrics/metrics.json` instead of computing values
in-place).

When a future ADR amends the metrics schema (e.g. adding a new field
beyond `value`/`unit`/`source`/`measured_at`/`hardware`), it carries
`Supersedes: ADR-016 (partial)` and the present file moves to
`Partially superseded by ADR-NNN` with a one-line pointer.
