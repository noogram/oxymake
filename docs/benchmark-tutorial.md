# Benchmark Tutorial

Run the full OxyMake DAG resolution benchmark in ~15 minutes.

## Prerequisites

```bash
# Build the release binary (required)
cargo build --release

# Verify the binary exists
./target/release/oxymake --version

# Python 3 is required for workload generation
python3 --version
```

**Optional tools:**

| Tool | Purpose | Install |
|------|---------|---------|
| `hyperfine` | Statistical timing (warmup, median, stddev) | `cargo install hyperfine` |
| `snakemake` | Performance comparison | `pip install snakemake` |
| `graphviz` | DAG visualization (`ox dag --format dot`) | `brew install graphviz` / `apt install graphviz` |

## Step 1: Generate Synthetic Workflows

The generator creates matching Oxymakefile.toml and Snakefile pairs at each scale.
Each workflow has a 3-level DAG: generate → process → merge.

```bash
# Generate workloads at 1K, 5K, and 10K jobs
python3 benchmark/perf/generate.py --sizes 1000,5000,10000 \
    --outdir benchmark/perf/workloads
```

This creates:

```
benchmark/perf/workloads/
├── scale_1000/
│   ├── Oxymakefile.toml    # 1,001 jobs (500 items × 2 rules + merge)
│   └── Snakefile
├── scale_5000/
│   ├── Oxymakefile.toml    # 5,001 jobs
│   └── Snakefile
└── scale_10000/
    ├── Oxymakefile.toml    # 10,001 jobs
    └── Snakefile
```

**Tip:** Start with 1K and 5K to verify everything works. Add 10K and 50K for
the full picture once you're comfortable.

## Step 2: Launch the Dashboard (Optional)

The web dashboard shows live execution progress, DAG structure, and job status.
You can watch it while benchmarks run.

```bash
# Start the dashboard in the background on port 9876
ox dashboard --port 9876 &
DASHBOARD_PID=$!

# Open in browser
open http://127.0.0.1:9876  # macOS
# xdg-open http://127.0.0.1:9876  # Linux
```

The dashboard reads from `.oxymake/state.db` and auto-refreshes via
server-sent events. You'll see the DAG populate as `ox plan` runs.

**Dashboard API endpoints** you can query during the benchmark:

```bash
curl http://127.0.0.1:9876/api/status     # Workflow status summary
curl http://127.0.0.1:9876/api/dag         # DAG structure (nodes + edges)
curl http://127.0.0.1:9876/api/jobs        # Job list
curl http://127.0.0.1:9876/api/stats/rules # Per-rule statistics
```

## Step 3: Run the Benchmark

The benchmark measures DAG resolution time only (`ox plan`), not execution.
This isolates the planner's performance from I/O and compute overhead.

```bash
# Run all sizes (1K, 5K, 10K, 50K)
bash benchmark/perf/run.sh

# Or run specific sizes
bash benchmark/perf/run.sh 1000 5000

# Or use the just recipe
just benchmark
```

**With dashboard monitoring** (starts dashboard in background automatically):

```bash
bash benchmark/perf/run.sh --dashboard
```

## Step 4: Compare with Snakemake (If Installed)

If `snakemake` is on your PATH, the benchmark automatically runs
`snakemake --dryrun` at each scale and produces a comparison table.

```bash
pip install snakemake  # install if needed
bash benchmark/perf/run.sh 1000 5000 10000
```

The results include a comparison table showing the speedup ratio:

```
## Comparison

| Jobs | Oxymake (s) | Snakemake (s) | Speedup |
|-----:|------------:|--------------:|--------:|
| 1000 |      0.0234 |        0.8500 |  36.3x  |
| 5000 |      0.1050 |        4.2000 |  40.0x  |
```

## Step 5: Record and Interpret Results

### Saved Results

After each run, results are written to two locations:

- **`benchmark/perf/results.md`** — Latest run (overwritten each time)
- **`benchmark/perf/results-TIMESTAMP.md`** — Timestamped archive copy

> **Note:** these `perf/` outputs are a **developer convenience and are
> git-ignored** — they are not a public numeric truth. The single benchmark of
> record cited by the paper and README is
> [`bench/snakemake-vs-oxymake/RESULTS.md`](../bench/snakemake-vs-oxymake/RESULTS.md),
> regenerated by `bash bench/snakemake-vs-oxymake/run.sh`.

### Interpreting the Numbers

| Metric | What It Tells You |
|--------|-------------------|
| **Median** | The representative timing (most stable across runs) |
| **Min** | Best-case performance (hot caches, no contention) |
| **Max** | Worst-case (GC pauses, system load) |
| **Stddev** | Measurement noise — lower is better for reproducibility |
| **Speedup** | How many times faster OxyMake is vs Snakemake |

**What to look for:**

- **Linear scaling:** 10K should take ~2× as long as 5K. Superlinear growth
  indicates algorithmic issues.
- **Low stddev:** If stddev is >20% of median, system noise is high. Close other
  applications and re-run.
- **Speedup consistency:** The speedup ratio should be roughly constant across
  scales. A ratio that drops at higher scales suggests OxyMake's planner
  doesn't scale as well as Snakemake's at that range.

### Screenshot the Dashboard

While the benchmark is running (or after), capture the dashboard state:

1. Open http://127.0.0.1:9876 in your browser
2. The DAG view shows the workflow structure at the current scale
3. The status panel shows job counts and timing

### Cleanup

```bash
# Stop the dashboard if running in background
kill $DASHBOARD_PID 2>/dev/null

# Remove generated workloads (optional)
rm -rf benchmark/perf/workloads
```
